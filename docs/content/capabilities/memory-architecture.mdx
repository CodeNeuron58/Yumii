## What v0.2.0 ships

Yumii v0.2.0 ships **two layers of persistent memory**, both backed by
local SQLite at `~/.yumii/memory/`:

| File | Backed by | Holds |
|------|-----------|-------|
| `yumii.db` | `aiosqlite` + `memory_db.py` | Session metadata (`sessions`, `session_summaries`) |
| `checkpoints.db` | `langgraph.checkpoint.sqlite.aio.AsyncSqliteSaver` | LangGraph conversation history, keyed by `thread_id` = session UUID |
| `store.db` | `langgraph.store.sqlite.AsyncSqliteStore` | User facts (preferences, identity, habits, goals) under namespace `("facts",)` |

`engine.py` opens `checkpoints.db` once and keeps the connection
alive for the lifetime of the process ‚Äî `AsyncSqliteSaver.from_conn_string`
is a short-lived context manager and would GC-close the connection
on the first turn otherwise. `MemoryManager` lazily opens `store.db`
on first fact read/write.

## Sessions

Each browser tab gets its own UUID session. The session is created on
WebSocket connect (or resumed via the `session_select` JSON command)
and lives in `sessions` (`id`, `name`, `created_at`, `last_active_at`,
`message_count`, `is_archived`). On every successful LLM turn the
engine bumps `last_active_at` via `session_manager.update_session_activity`.

The LangGraph `thread_id` for `graph_app.ainvoke(...)` is the active
session id, so checkpoints are namespaced per conversation. Restarting
the server does **not** clear conversation history ‚Äî your last chat is
exactly where you left it.

CLI:
* `/chat` ‚Äî TUI session picker (resume, new, delete)
* `/resume` ‚Äî resume most recent session
* `/sessions` ‚Äî list all saved sessions
* `/name <name>` ‚Äî rename the active session (browser only)

REST:
* `GET /api/sessions`
* `POST /api/sessions` (optional `?name=`)
* `POST /api/sessions/{id}/resume`
* `DELETE /api/sessions/{id}`

## Long-term memory (user facts)

After every conversation turn, `memory_manager.extract_facts_from_messages(...)`
runs a **fire-and-forget** background task: a cheap LLM pass
(Groq `llama-3.1-8b-instant` by default, OpenAI `gpt-4o-mini` or
`claude-3-5-sonnet-latest` as fallbacks) analyses the user/assistant
exchange and returns a JSON list of atomic facts:

```json
{
  "facts": [
    {"fact": "user is vegetarian", "category": "preference", "confidence": 0.9},
    {"fact": "user lives in Berlin", "category": "identity",    "confidence": 0.95}
  ]
}
```

Each fact is deduplicated against existing memory (substring match)
before being stored under the `("facts",)` namespace. The categories
are: `preference`, `identity`, `habit`, `relationship`, `goal`, `general`.

On every new or resumed session, the engine pre-loads
`memory_manager.get_facts_raw()` and injects them into the
`MainState` of the LangGraph run. `chat_node` formats them as
bullet points and appends to the personality system prompt at agent
build time. The agent is cached per `(personality, hash(facts))` ‚Äî
editing a fact invalidates the cache and the next turn picks up the
new memory.

CLI:
* `/memory` ‚Äî TUI fact browser / editor / deleter
* `/forget` ‚Äî wipe **all** facts (sessions are kept)

REST:
* `GET /api/facts`
* `PUT /api/facts/{id}` (body: `{"fact": "..."}`)
* `DELETE /api/facts/{id}`

## Why SQLite, not Postgres or a vector store?

* **Local-first.** No cloud roundtrip for memory, no API key for storage.
* **Tiny corpus.** The fact store is bounded by what one user can
  possibly say in a few years of conversation ‚Äî well under 10 000 rows.
  Substring deduplication is more than fast enough.
* **LangGraph native.** `AsyncSqliteStore` and `AsyncSqliteSaver` are
  the project's reference implementations, so the integration is
  ~30 lines, not 300.
* **Embeddings later.** If retrieval quality (not storage size) ever
  becomes the bottleneck, swapping the fact lookup for a vector
  similarity search is a single-function change in `MemoryManager`.
