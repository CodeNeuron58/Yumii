## Status: Planned for v2.0

MCP support is on the Yumi roadmap but is **not implemented in v1.0.0**.
The source modules and CLI subcommand referenced by older drafts of this
page have been removed from the codebase.

The current architecture (`src/yumi/core/engine.py`) is a single-loop
voice → LLM → TTS pipeline. It does not yet expose a stdio MCP server,
a `ToolContract` registry, or a `yumi mcp start` command.

**What to do today:** if you need a tool-using agent, use a third-party
MCP client (Claude Desktop, Cursor) and wire it to your own services.
Yumi's value in v1 is the **conversational loop and animated avatar**,
not the integration surface.

**For v2:** see `ROADMAP.md` — v2.0 (planned Q3–Q4 2026) will introduce
a Triage / Planner / Synthesizer agent loop with a `ToolContract`
protocol and MCP server compatibility.
</ContentCard>

<ContentCard>
## Why this is on hold

MCP is a transport for sharing tools with external LLM apps. Yumi v1
has one tool (`get_current_time`) and no general tool registry. Adding
an MCP server before the agent loop exists would be surface area
without a backend.

The intended order is:

1. **v1.0** — conversational loop, avatar, six personalities.
2. **v1.1** — persistent conversation log (SQLite) and basic user facts.
3. **v2.0** — Triage / Planner / Synthesizer agent loop with a
   `ToolContract` protocol. MCP server transport follows from that,
   because at that point there is a real tool registry to expose.

If you want to follow or contribute to v2, watch the
[GitHub releases page](https://github.com/CodeNeuron58/Yumi/releases)
or open an issue.
</ContentCard>
